# Modifying / Adding a Printer

A "printer" is a machine model (e.g. V-Core 4 IDEX). Each lives under
`configuration/printers/<id>/` and is **auto-discovered** by the wizard. A printer has two
halves:

1. **Data** â€” `printer-definition.json` + the machine's `.cfg` files (bed sizes, macros).
   Adding these needs **no rebuild**.
2. **A generator template** â€” `configurator/src/templates/printers/<name>.ts`, named by the
   `template` field. Reusing an existing template needs no rebuild; writing a **new**
   template means recompiling the configurator (see Â§5).

## 1. What a printer folder contains

Example `configuration/printers/v-core-4/`:

| File | Purpose |
|---|---|
| `printer-definition.json` | Metadata the wizard reads: sizes, speed limits, default hardware, and which `template` generates the config. Validated against `printers/printer-definition.schema.json`. |
| `<id>.cfg` | The base machine config: `[printer]` kinematics + limits, `[heater_bed]`, `[bed_mesh]` split. Header says **DO NOT EDIT** (bundled). |
| `300.cfg` / `400.cfg` / `500.cfg` | Per-**size** overrides (position_max, mesh areaâ€¦). The generator `[include]`s the one matching the chosen size. |
| `macros.cfg` | Printer-specific macro tweaks. |
| `<id>.png` | Image shown in the printer picker. |

Shared files one level up: `printers/base.cfg` (common includes),
`printers/initial-setup.cfg`, and the two schema files.

## 2. `printer-definition.json` fields

Required: `name`, `description`, `manufacturer`, `documentationLink`,
`driverCountRequired`, `image`, `template`, `defaults`, `speedLimits`.

| Field | Meaning |
|---|---|
| `template` | **The `.ts` file in `templates/printers/` that generates the config** (e.g. `"v-core-4.ts"`). This is the link between data and code. |
| `driverCountRequired` | How many drivers the machine needs â€” the wizard validates the chosen board has enough. |
| `sizes` | Map of build-volume presets (`"300": {x,y,z}`). Each key must have a matching `<size>.cfg`. |
| `bedMargin` | Unprobeable bed margins `{x:[left,right], y:[front,back]}` for mesh/parking math. |
| `speedLimits.basic` / `.performance` | Two accel/velocity profiles; the wizard offers "performance mode". |
| `defaults` | Pre-selected hardware so the wizard starts sensible: `board`, `toolheads[]` (extruder, toolboard, hotend, fans, probe, endstops per carriage), and `rails[]` (per-axis driver, stepper, current, rotation distance, homing speed). |
| `kinematics` | Motion type when not implied by the template. |
| `releaseDate` | Cosmetic. |

The `defaults.toolheads[]` entries reference **ids from other `configuration/` libraries**
(`extruders/`, `hotends/`, `toolboard/`, `z-probe/`, `endstops`) and from board ids. Those
referenced parts must exist or the wizard can't resolve the default.

## 3. The generator template (`templates/printers/<name>.ts`)

A template is an `async` function returning the text of `Ratical.cfg`:

```ts
export const template = async (config: PrinterConfiguration, helper: KlipperConfigHelper) => `
# WARNING. THIS FILE IS GENERATED BY THE RATICAL CONFIGURATOR...
${helper.renderBoards()}
${helper.renderBase()}
[include Ratical/printers/v-core-4/v-core-4.cfg]
[include Ratical/printers/v-core-4/${config.size.x}.cfg]
${helper.renderExtruder()}
${helper.renderHotend()}
${helper.renderInputShaper(config.size)}
${helper.renderMotorSections()}
${helper.renderSpeedLimits()}
...`;
```

- `config` â€” the user's wizard choices (typed by `zods/printer-configuration`).
- `helper` (`KlipperConfigHelper`, in `server/helpers/klipper-config.ts`) â€” renders each
  section: `renderBoards`, `renderBase`, `renderExtruder`, `renderHotend`,
  `renderInputShaper`, `renderMotorSections`, `renderServoEnableDelay`, `renderSpeedLimits`,
  `getMacroTravelSpeed`, etc. **Use the helper** rather than hand-writing sections â€” it keeps
  board/toolboard/driver wiring consistent.
- Everything is a template literal, so you interpolate `config.size.x`, chosen hotend, etc.

IDEX / hybrid variants (`v-core-4-idex.ts`, `-hybrid.ts`) are separate templates that add a
`dual_carriage` / second-toolhead section on top of the same base.

## 4. Step-by-step: add a printer that reuses an existing template

Best path when your machine is close to a bundled one (no rebuild needed).

1. **Copy the closest printer folder:**
   ```bash
   cp -r configuration/printers/v-core-4 configuration/printers/my-printer
   cd configuration/printers/my-printer && mv v-core-4.cfg my-printer.cfg
   ```
2. **Edit `printer-definition.json`:** set `name`, `description`, `manufacturer`, `image`,
   `sizes`, `bedMargin`, `speedLimits`, `defaults`, and keep or change `template`
   (e.g. keep `"v-core-4.ts"` if the kinematics match).
3. **Edit `my-printer.cfg`** and the size cfgs for your bed limits / hardware.
4. **If the template hard-codes the old id** in `[include Ratical/printers/v-core-4/...]`
   lines, you need your own template (Â§5) or a template that derives the path from
   `config.printer.id`. Check the template you chose.
5. **Validate + install:**
   ```bash
   python3 -c "import json;json.load(open('printer-definition.json'));print('ok')"
   cd ~/ratical && ./install.sh 30
   ```
   Reopen `/configure/`; pick your printer and run the wizard to generate `Ratical.cfg`.

## 5. Add a printer that needs a NEW template (requires rebuild)

1. Copy `templates/printers/v-core-4.ts` â†’ `templates/printers/my-printer.ts`; adjust the
   `[include â€¦]` paths and section logic.
2. Point your `printer-definition.json` `template` at `"my-printer.ts"`.
3. **Rebuild the configurator on a dev machine** (never on the Pi â€” 1 GB OOMs):
   ```bash
   scripts/build-configurator.sh
   git add configurator/src/build configurator/src/bin configurator/src/templates
   git commit -m "feat: add my-printer template"
   ```
4. On the Pi: `cd ~/ratical && git pull && ./install.sh 20 30`.

See [modifying/configurator.md](./configurator.md) for the rebuild workflow in detail.

## 6. Gotchas

- **`template` names a file that must exist and be built into the app.** A definition can be
  pure data, but if it names a template you haven't compiled, generation fails.
- **`sizes` keys must have matching `<size>.cfg` files** â€” the template `[include]`s
  `${config.size.x}.cfg`.
- **`defaults` reference other libraries** (extruders/hotends/toolboards/boards). Those ids
  must exist.
- **Bundled cfgs say DO NOT EDIT** because a regenerate/update can rewrite the surrounding
  config â€” persistent user changes belong in the top-level `printer.cfg`.
- **Kalico minimums:** e.g. `split_delta_z` must be â‰Ą `0.01` (step 36 fixes this
  automatically if you copy an older value).
